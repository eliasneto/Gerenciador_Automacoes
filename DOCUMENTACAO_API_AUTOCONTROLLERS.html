Documentacao Oficial da API - AutoControllers

Este documento descreve a camada de API separada do AutoControllers, criada para integracoes externas sem alterar o fluxo web atual do sistema.

Resumo rapido:
A API do AutoControllers usa autenticacao por token do tipo Bearer, respeita os grupos e permissoes ja existentes no sistema e expõe endpoints para autenticação, consulta de modulos, automacoes, execucoes e documentacoes publicadas.

1. Informacoes gerais

ItemValor
Base URL localhttp://127.0.0.1:8190/api/
Formato principalapplication/json
AutenticacaoToken Bearer
SegurancaPermissoes alinhadas com os grupos do sistema
Login web atualPermanece funcionando em paralelo
Importante: esta API nao substitui as telas HTML. Ela foi criada em uma camada separada, no app api, para evitar impacto no sistema web ja existente.

2. Autenticacao

O fluxo esperado e:

  1. enviar username e password para obter um token
  2. receber o token no campo access_token
  3. usar este token no header Authorization: Bearer TOKEN

Header padrao

Authorization: Bearer SEU_TOKEN_AQUI

Tambem e aceito:

X-API-Token: SEU_TOKEN_AQUI

3. Padrao de resposta

Em geral, a API responde com JSON no formato:

{
  "ok": true,
  "results": []
}

Ou, em caso de erro:

{
  "ok": false,
  "error": "Mensagem do erro"
}

4. Codigos de status mais usados

CodigoUso
200Operacao concluida com sucesso
201Recurso ou execucao criada com sucesso
400Payload invalido, dados faltando ou regra de negocio violada
401Credenciais invalidas ou token ausente/invalido
403Usuario autenticado, mas sem permissao para o recurso
404Recurso nao encontrado

5. Paginacao

Os endpoints de lista agora aceitam paginação por query string.

6. Endpoints

GET/api/health/

Endpoint simples para validar se a camada de API esta respondendo.

Resposta de sucesso

{
  "ok": true,
  "service": "autocontrollers-api",
  "timestamp": "2026-03-29T10:15:00-03:00"
}

POST/api/auth/token/

Realiza autenticacao com usuario e senha e devolve o token de acesso.

Exemplo de requisicao

POST /api/auth/token/
Content-Type: application/json

{
  "username": "admin",
  "password": "admin123"
}

Resposta de sucesso

{
  "ok": true,
  "token_type": "Bearer",
  "access_token": "TOKEN_GERADO_PELO_SISTEMA",
  "user": {
    "id": 1,
    "username": "admin",
    "is_superuser": true
  }
}

Erros possiveis

StatusQuando aconteceExemplo
400Username ou password nao enviados{"ok": false, "error": "Informe username e password."}
400JSON invalido{"ok": false, "error": "Payload JSON inválido."}
401Credenciais incorretas{"ok": false, "error": "Credenciais inválidas."}

POST/api/auth/token/revoke/

Revoga o token atual.

Exemplo de requisicao

POST /api/auth/token/revoke/
Authorization: Bearer TOKEN_GERADO_PELO_SISTEMA

Resposta de sucesso

{
  "ok": true,
  "message": "Token revogado com sucesso."
}

Erros possiveis

StatusQuando aconteceExemplo
401Token ausente, inativo, expirado ou invalido{"ok": false, "error": "Token de API inválido ou ausente."}

7. Erros padrao da autenticacao por token

Quase todos os endpoints protegidos podem retornar este erro quando o token nao for enviado corretamente:

{
  "ok": false,
  "error": "Token de API inválido ou ausente."
}

8. Erros padrao de permissao

Quando o usuario esta autenticado, mas nao tem acesso ao recurso, a API retorna 403.

Exemplo

{
  "ok": false,
  "error": "Você não possui acesso a este módulo."
}

9. Erros padrao de recurso nao encontrado

Quando modulo, automacao ou execucao nao forem encontrados, a API retorna 404.

Exemplos

{
  "ok": false,
  "error": "Módulo não encontrado."
}
{
  "ok": false,
  "error": "Automação não encontrada."
}
{
  "ok": false,
  "error": "Execução não encontrada."
}

10. Exemplo completo de uso

Fluxo simples de integracao:

  1. fazer login por token
  2. listar automacoes
  3. executar uma automacao
  4. consultar a execucao ate finalizar

Passo 1 - obter token

POST /api/auth/token/
{
  "username": "admin",
  "password": "admin123"
}

Passo 2 - listar automacoes

GET /api/automacoes/
Authorization: Bearer TOKEN_GERADO_PELO_SISTEMA

Passo 3 - executar

POST /api/automacoes/comercial/pesquisar-youtube/executar/
Authorization: Bearer TOKEN_GERADO_PELO_SISTEMA

Passo 4 - acompanhar

GET /api/execucoes/23/
Authorization: Bearer TOKEN_GERADO_PELO_SISTEMA

11. Observacoes importantes

  • esta primeira versao da API e um MVP funcional
  • ela nao substitui as telas existentes
  • ela reaproveita as regras de grupos e permissao do sistema
  • ela ainda pode evoluir com paginacao, criacao de multiplos tokens e documentacao automatizada
Atencao: como a API utiliza os grupos atuais do sistema, um problema de permissao na interface web normalmente se refletira tambem na API. Isso e bom para manter coerencia, mas precisa ser lembrado durante testes de integracao.

12. Checklist antes de usar em integracoes

  • rodar manage.py migrate para criar a tabela de token
  • validar se o usuario de integracao esta ativo
  • confirmar os grupos do usuario
  • testar o endpoint /api/health/
  • testar o login em /api/auth/token/
  • testar consulta basica com /api/me/
Regra final para nao errar:
Primeiro obtenha o token, depois valide as permissoes do usuario com /api/me/, e so entao comece a consumir automacoes, execucoes e documentacoes. Isso reduz muito erro de integracao e acelera o diagnostico quando algo falhar.
