Guia Oficial - Como Criar uma Automacao no AutoControllers

Este documento explica, do inicio ao fim, como criar uma automacao compativel com o AutoControllers na arquitetura atual do sistema.

Resumo rapido:
Hoje o AutoControllers roda com web + worker + MySQL em Docker. A interface registra a execucao, a fila controla a concorrencia e o worker executa as automacoes em segundo plano. Para uma automacao funcionar bem, ela precisa respeitar o contrato da funcao executar(...), gravar os resultados em output_dir e ser compativel com container quando for publicada em producao.

1. Visao geral da arquitetura atual

O sistema foi organizado para separar interface, fila e execucao.

ComponenteResponsabilidade
webRecebe o clique do usuario, registra a execucao, aplica regras do sistema, mostra logs, status e arquivos.
workerConsome a fila e executa as automacoes em background.
MySQLArmazena usuarios, automacoes, execucoes, fila, documentacao e auditorias.

No ambiente Docker, o sistema sobe na porta 8190.

http://127.0.0.1:8190
Importante: o clique no frontend nao executa diretamente a automacao dentro da pagina. O sistema registra a execucao no banco e o worker e quem pega essa execucao para processar.
Acesso inicial em Docker: o container web pode criar ou atualizar automaticamente o superusuario inicial com base nas variaveis DJANGO_SUPERUSER_USERNAME, DJANGO_SUPERUSER_EMAIL e DJANGO_SUPERUSER_PASSWORD. Isso ajuda a liberar rapidamente a area administrativa para cadastrar automacoes em ambientes novos.

2. Onde salvar o arquivo da automacao

Toda automacao deve ser salva dentro da pasta automacoes do setor correspondente.

SetorPasta correta
Comercialcomercial/automacoes/
Financeirofinanceiro/automacoes/
TIti/automacoes/

Exemplos de arquivos validos:

comercial/automacoes/pesquisar_youtube.py
financeiro/automacoes/conciliar_pagamentos.py
ti/automacoes/processar_inventario.py
Regra pratica: o nome do arquivo faz parte do executor_path. Se o arquivo estiver na pasta errada, ou se o caminho informado no cadastro estiver incorreto, a automacao nao sera importada pelo sistema.

3. Estrutura obrigatoria da automacao

Cada arquivo de automacao precisa expor uma funcao chamada executar.

def executar(
    input_path=None,
    input_paths=None,
    attachments=None,
    output_dir=None,
    should_stop=None,
    log=None,
    parametros=None,
    parametros_json=None,
):
    ...

Significado de cada parametro

  • input_path: primeiro arquivo principal
  • input_paths: lista com os arquivos principais enviados
  • attachments: lista com os arquivos secundarios
  • output_dir: pasta onde a automacao deve salvar os arquivos de saida
  • should_stop(): funcao para verificar se o usuario pediu parada
  • log('mensagem'): envia mensagens para o log exibido no sistema
  • parametros: observacao textual informada pelo usuario na execucao
  • parametros_json: observacao convertida para JSON, quando aplicavel

4. Como o sistema entrega os arquivos para a automacao

Os arquivos fisicamente ficam em media/, mas a automacao nao deve sair procurando arquivos em uma pasta fixa do projeto.

O jeito correto de trabalhar e usar os parametros recebidos na funcao:

  • input_paths para arquivos principais
  • attachments para arquivos secundarios
  • output_dir para os arquivos gerados

Exemplo:

input_paths = input_paths or ([] if input_path is None else [input_path])
attachments = attachments or []

arquivo_principal_1 = input_paths[0] if input_paths else None
arquivo_principal_2 = input_paths[1] if len(input_paths) > 1 else None
Regra do sistema: o caminho exato do arquivo ja chega pronto na sua automacao. Voce nao precisa montar manualmente caminhos dentro de media/ para abrir os arquivos recebidos.

5. Como salvar o resultado corretamente

Todo arquivo final que o usuario devera baixar precisa ser salvo dentro de output_dir.

Exemplo:

from pathlib import Path

saida = Path(output_dir) / 'resultado.csv'
saida.write_text('ok', encoding='utf-8')

Se o resultado nao for salvo em output_dir, o sistema nao consegue exibir esse arquivo no frontend.

6. Como a fila funciona

O sistema possui controle de concorrencia. Isso significa que nem toda execucao comeca imediatamente.

  1. O usuario clica em executar uma automacao
  2. O sistema cria o registro da execucao
  3. Se houver vaga, a execucao pode comecar na hora
  4. Se o limite estiver ocupado, a execucao fica em Pendente
  5. O worker monitora a fila e inicia novas execucoes quando houver espaco

Status possiveis da execucao:

  • Pendente
  • Executando
  • Sucesso
  • Erro
  • Interrompida
Boa pratica: escreva automacoes pensando que elas podem esperar na fila antes de comecar. Evite qualquer logica que dependa do horario exato do clique do usuario se isso nao for estritamente necessario.

7. Como respeitar a parada da execucao

Quando o usuario pede parada, o sistema registra essa solicitacao. A sua automacao precisa verificar isso durante o processamento.

Exemplo:

if should_stop:
    should_stop()

O ideal e chamar should_stop():

  • antes de loops grandes
  • antes de consumir muitos arquivos
  • antes de chamadas externas demoradas
  • antes de escrever grandes volumes de resultado

8. Como escrever logs de forma correta

Use log() para enviar mensagens ao monitor da execucao.

if log:
    log('Iniciando automacao')
    log(f'Arquivo principal recebido: {arquivo_principal_1.name}')

Tambem e possivel usar print() em alguns fluxos, mas a forma mais clara para o sistema continua sendo log().

Boas praticas de log

  • informe o inicio da execucao
  • mostre o nome dos arquivos recebidos
  • registre etapas importantes
  • registre o nome do arquivo final salvo
  • evite logar dados sensiveis

9. Exemplo minimo de automacao compativel

from pathlib import Path


def executar(
    input_path=None,
    input_paths=None,
    attachments=None,
    output_dir=None,
    should_stop=None,
    log=None,
    parametros=None,
    parametros_json=None,
):
    input_paths = input_paths or ([] if input_path is None else [input_path])
    attachments = attachments or []
    output_dir = Path(output_dir)
    output_dir.mkdir(parents=True, exist_ok=True)

    if not input_paths:
        raise ValueError('Envie pelo menos um arquivo principal.')

    arquivo_principal_1 = input_paths[0]
    arquivo_principal_2 = input_paths[1] if len(input_paths) > 1 else None

    if log:
        log('Iniciando automacao')
        log(f'Arquivo principal 1: {arquivo_principal_1.name}')
        if arquivo_principal_2:
            log(f'Arquivo principal 2: {arquivo_principal_2.name}')
        for anexo in attachments:
            log(f'Arquivo secundario: {anexo.name}')

    if should_stop:
        should_stop()

    arquivo_saida = output_dir / 'resultado.txt'
    arquivo_saida.write_text('Automacao executada com sucesso.', encoding='utf-8')

    if log:
        log(f'Resultado salvo em: {arquivo_saida.name}')

    return {
        'message': 'Automacao concluida com sucesso.'
    }

10. Como cadastrar a automacao no sistema

  1. Entre com usuario com acesso a area Administrador
  2. No menu lateral, entre em Administrador
  3. Clique em Criar Automacao
  4. Preencha os campos principais
  5. Salve

Campos principais do cadastro

  • Nome da automacao
  • Identificador
  • Descricao
  • Executor Python
  • Setor ou setores
  • Aceita arquivo principal
  • Quantidade maxima de arquivos principais
  • Aceita anexos
  • Ativa

Como montar o executor_path

Se o arquivo estiver salvo em:

financeiro/automacoes/meu_robo.py

e a funcao for executar, entao o valor no cadastro deve ser:

financeiro.automacoes.meu_robo.executar
Regra simples:
setor/automacoes/nome_do_arquivo.py
se transforma em
setor.automacoes.nome_do_arquivo.executar

11. Exemplo pratico do projeto

O projeto possui uma automacao exemplo para referencia tecnica:

comercial/automacoes/pesquisar_youtube.py

Ela segue este fluxo:

  1. recebe um tema.txt como arquivo principal
  2. le o tema de pesquisa
  3. consulta o YouTube
  4. gera um arquivo resultado_youtube.csv
  5. salva o resultado em output_dir
  6. disponibiliza o download no frontend

12. Como a automacao aparece para o usuario final

Depois de cadastrada e ativa, a automacao passa a aparecer dentro do modulo correspondente.

O usuario pode:

  • abrir o modal da automacao
  • anexar arquivos
  • informar observacoes
  • executar
  • acompanhar logs
  • baixar o arquivo final, se houver saida

13. O que roda bem em Docker

No ambiente atual, o sistema foi preparado para trabalhar muito bem com automacoes baseadas em web, arquivos e integracoes.

Boas escolhas para novas automacoes

  • Playwright para automacao web headless
  • requests e httpx para integracoes
  • pandas e openpyxl para planilhas
  • pdfplumber e pypdf para PDFs
  • beautifulsoup4 para leitura de HTML
Recomendacao principal: para novas automacoes web dentro do Docker, prefira Playwright. Hoje ele e a opcao mais alinhada com a arquitetura do projeto.

14. O que nao e ideal em Docker

Automações desktop visuais nao sao a melhor opcao dentro de um container de servidor.

Exemplo de biblioteca que exige mais cuidado:

  • pyautogui

Motivos:

  • depende de ambiente grafico
  • container nao representa a tela real do usuario
  • mouse e teclado do container nao sao o desktop do operador
Conclusao pratica: se a automacao for web, use Playwright. Se for desktop tradicional, o ideal e uma esteira separada com worker apropriado para GUI, maquina virtual ou servidor com sessao grafica controlada.

15. Bibliotecas ja preparadas no projeto

O projeto ja foi preparado com bibliotecas comuns de automacao, leitura de arquivos e integracao. Mesmo assim, a automacao deve importar somente o que realmente usar.

Exemplos de bibliotecas presentes na esteira atual:

16. Regras de publicacao para automacoes novas

  • grave sempre os resultados em output_dir
  • nao dependa de caminhos fixos fora do contrato da execucao
  • registre logs claros
  • trate validacao de arquivo principal
  • respeite a parada com should_stop()
  • retorne uma mensagem final sempre que possivel
  • prefira automacoes compativeis com Docker para producao

17. O que nao deve ir para producao

O projeto ja teve automacoes de exemplo para demonstracao, mas a esteira atual foi ajustada para nao levar seeds de testes para producao.

Isso significa que novas automacoes publicadas devem ser somente automacoes reais do ambiente. Arquivos de exemplo podem existir no repositorio como referencia tecnica, mas nao devem ser tratados como cadastro padrao de producao.

18. Checklist antes de cadastrar uma automacao

  1. o arquivo esta na pasta correta do setor
  2. a funcao se chama executar
  3. o executor_path foi montado corretamente
  4. a automacao valida os arquivos recebidos
  5. a automacao usa output_dir para o resultado
  6. a automacao escreve logs suficientes
  7. a automacao respeita should_stop()
  8. a automacao e compativel com Docker ou foi desenhada para a esteira correta

20. Fluxo final da execucao

  1. O usuario abre o modulo
  2. Seleciona a automacao
  3. Envia os arquivos necessarios
  4. O sistema registra a execucao
  5. A fila decide se comeca agora ou se fica pendente
  6. O worker executa a rotina
  7. Os logs aparecem no monitor
  8. O arquivo salvo em output_dir aparece para download

21. Conclusao

Hoje o AutoControllers esta preparado para receber automacoes reais com uma arquitetura mais madura, separando interface, fila, banco e execucao.

Se voce seguir este guia, a automacao fica alinhada com o comportamento que o sistema espera em desenvolvimento e em producao.

Regra final para nao errar:
Salve o arquivo no setor certo, exponha a funcao executar(...), use os parametros entregues pelo sistema, grave a saida em output_dir e prefira automacoes web e de arquivos para rodar no Docker com mais seguranca.
