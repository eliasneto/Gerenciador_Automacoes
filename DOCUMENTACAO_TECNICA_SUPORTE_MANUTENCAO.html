Documentacao Tecnica - Suporte e Manutencao do AutoControllers

Este documento foi escrito para equipes de suporte, manutencao e evolucao do AutoControllers. O objetivo e permitir que uma pessoa assuma o sistema com seguranca, entendendo a arquitetura, os modulos, as rotas, as regras de negocio, os grupos de seguranca e a responsabilidade de cada arquivo importante.

Resumo executivo:
O AutoControllers e um sistema Django com arquitetura web + worker + MySQL, executado em Docker na porta 8190. A interface registra execucoes, a fila controla concorrencia, o worker processa automacoes em segundo plano, o modulo de documentacao centraliza consulta e publicacao de conhecimento operacional e uma camada separada de API expõe integracoes por token sem substituir o fluxo web atual.
Versao atual:
A release atual publicada no Git esta identificada como v1.0.0. Essa mesma versao aparece no menu lateral abaixo das informacoes do usuario e tambem no rodape da interface autenticada.

1. Arquitetura geral do sistema

ComponenteResponsabilidade
webAplicacao Django/Gunicorn. Recebe login, navegaçao, cadastros, execucoes, documentacao e admin.
workerProcessa a fila de automacoes pendentes e dispara o comando run_automation.
dbMySQL 8.4. Armazena usuarios, grupos, automacoes, execucoes, fila, documentos e auditorias.
mediaArquivos de entrada, anexos e saidas das automacoes.
static/staticfilesArquivos estaticos da interface e da administracao.

Entrada principal do sistema:

http://127.0.0.1:8190/
Fluxo tecnico de execucao:
Usuario clica em executar -> Django valida formulario e arquivos -> cria registro em AutomationExecution -> fila decide se executa ou aguarda -> worker monitora pendencias -> comando manage.py run_automation <id> importa a funcao Python da automacao e executa -> arquivos gerados em output_dir sao registrados no banco para download.

1.1 Identificacao de release

  • A versao funcional exibida ao usuario vem de config/settings.py, pela constante APP_VERSION.
  • Essa versao e entregue aos templates por core/context_processors.py.
  • O menu lateral e o rodape leem essa informacao em templates/base.html.
  • O Git usa tag anotada para marcar releases. A primeira tag oficial do projeto e v1.0.0.

3.3 Regras atuais

  • Usuario so ve modulo se estiver no grupo correspondente.
  • Usuario so ve indicadores do dashboard dos grupos de dashboard que possuir.
  • superuser ve tudo, mesmo sem grupo.
  • Documentacao Sistema fica acessivel a qualquer usuario autenticado.
  • Edicao da documentacao continua restrita ao grupo de documentacao.
  • A API respeita os mesmos grupos de modulo e administracao ja usados pela interface web.
Ponto importante para suporte:
Quando um usuario disser que "o modulo sumiu", quase sempre a causa esta em grupos de permissao no Django Admin, e nao em erro de rota ou template.

4. Regras de negocio centrais

  1. Cada automacao pertence a um ou mais setores no momento do cadastro administrativo.
  2. Execucoes respeitam fila e limite de concorrencia.
  3. Se a fila estiver lotada, a execucao entra em Pendente.
  4. Documentacao vinculada a automacao so aparece ao usuario se estiver Publicado.
  5. Ao editar um documento existente, ele volta para Rascunho.
  6. Somente uma documentacao pode ficar vinculada a uma mesma automacao.
  7. Documentos publicados sem vinculo com automacao podem aparecer em Sistema, Administracao, Comercial, Financeiro ou TI, conforme o setor de publicacao.
  8. Automações de exemplo nao devem ser tratadas como seeds de producao.

Regras de negocio da API

  • A API nao substitui o login web por sessao; ela roda em paralelo.
  • Autenticacao usa token do tipo Bearer.
  • Os tokens podem ser revogados sem afetar o login web do usuario.
  • Permissoes respeitam os mesmos grupos usados nas telas.
  • Endpoints de lista possuem paginação com page e per_page.
  • O endpoint de execucao reaproveita a mesma logica da fila do sistema web.
Observacao operacional:
Incidentes de API tendem a se concentrar em quatro pontos: token ausente/invalido, usuario sem grupo, modulo inacessivel ou migration da tabela APIToken ainda nao aplicada.

Modelos principais do core

  • AutomationBaseModel: base herdada pelos modelos de setor.
  • AutomationQueueSettings: define quantas execucoes simultaneas podem rodar.
  • AutomationExecution: historico, estado e metricas de cada execucao.
  • AutomationExecutionFile: arquivos anexados ou gerados em cada execucao.
  • AutomationAsset: arquivos salvos por automacao para reaproveitamento.

Rotas do core

  • /dashboard/

Regras de negocio do core

  • Se o limite de concorrencia estiver cheio, a nova execucao entra em fila.
  • Se o usuario pedir parada de uma execucao pendente, ela vira Interrompida sem rodar.
  • Se o usuario pedir parada de uma execucao em andamento, o sistema marca interromper_solicitado e tenta enviar sinal ao processo.
  • Resultados so aparecem ao usuario se forem gravados em output_dir e registrados em AutomationExecutionFile.
  • Durante a execucao, o processo agora grava memoria atual, pico de memoria, CPU de usuario, CPU de sistema e data da ultima atualizacao.

13.2 Pontos para verificar quando algo falhar

  1. O container db esta de pe?
  2. O container web subiu sem erro de migrate?
  3. O container worker esta ativo?
  4. Se o problema for na API, a migration do app api foi aplicada?
  5. Se o problema for no monitoramento por execucao, a migration core 0005 foi aplicada?
  6. O usuario tem o grupo certo?
  7. A automacao esta marcada como ativa?
  8. O executor_path realmente aponta para um arquivo e uma funcao existentes?
  9. A automacao gravou resultado em output_dir?
  10. O documento esta publicado ou ainda esta em rascunho?

15. Regras para manutencao segura

  • Nao alterar grupos padrao sem revisar core/security.py e menu lateral.
  • Nao mudar executor_path sem garantir que a funcao exista e seja importavel.
  • Nao publicar documentacao em nome do usuario final sem validar o status.
  • Nao assumir que automacao desktop com pyautogui vai funcionar bem no Docker.
  • Ao adicionar novo setor, revisar pelo menos: modelo, rotas, views, registry, seguranca, dashboard e menu.
  • Ao alterar filas, validar o comportamento do worker e da configuracao de concorrencia.
  • Ao alterar a API, preservar a separacao da camada /api/ e evitar mexer nas views HTML sem necessidade.
  • Ao alterar paginação da API, validar page, per_page, has_next e has_previous.

16. Extensao futura do sistema

Se for necessario criar um novo modulo setorial, os pontos minimos de extensao costumam ser:

  1. Criar novo app Django.
  2. Criar modelo herdando de AutomationBaseModel.
  3. Criar rotas e views especializadas usando ModuleHomeView, StartAutomationView, StopAutomationView e ExecutionLogView.
  4. Registrar o setor em core/sector_registry.py.
  5. Garantir grupos em core/security.py e sincronizacao dos grupos.
  6. Atualizar menu lateral e dashboard.

16.1 Extensao futura da API

  1. Manter novos endpoints dentro do app api.
  2. Reaproveitar regras de permissao do core/security.py.
  3. Preferir reaproveitar servicos de negocio existentes antes de criar logica duplicada.
  4. Documentar qualquer endpoint novo no arquivo DOCUMENTACAO_API_AUTOCONTROLLERS.html.
  5. Se houver adocao futura de Swagger/OpenAPI, preservar compatibilidade da rota base /api/.
Conclusao:
Para suporte e manutencao, os arquivos mais sensiveis do projeto hoje sao config/settings.py, config/urls.py, core/services.py, core/security.py, core/sector_registry.py, documentacao/views.py, documentacao/services.py, api/auth.py, api/views.py, docker/entrypoint.sh e docker-compose.yml. Se esses pontos estiverem entendidos, a maior parte dos incidentes do sistema fica muito mais facil de diagnosticar.
