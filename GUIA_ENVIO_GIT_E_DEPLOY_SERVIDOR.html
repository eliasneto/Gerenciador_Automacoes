Guia Oficial - Git, GitHub e Deploy do AutoControllers

Este documento descreve o fluxo completo para versionar o projeto no Git, enviar para o GitHub, preparar um servidor Linux e subir o AutoControllers em producao com Docker.

Resumo rapido:
O fluxo recomendado e: preparar o repositório local, criar o repositório no GitHub, fazer o primeiro push, acessar o servidor Linux, instalar Docker e Docker Compose, configurar o arquivo de ambiente, clonar o projeto e subir os containers com docker compose up --build -d ou docker-compose up --build -d.
Versao atual da release:
A release inicial publicada do AutoControllers esta marcada no Git como v1.0.0. Essa mesma versao aparece na interface do sistema, abaixo das informacoes de login no menu lateral e tambem no rodape.

6.1 Como criar e publicar uma tag de release

Quando voce quiser marcar uma versao oficial, use uma tag anotada. O ideal e que a tag do Git acompanhe a versao exibida no sistema.

Linux

git tag -a v1.0.0 -m "Initial stable release"
git push origin v1.0.0

Windows PowerShell

git tag -a v1.0.0 -m "Initial stable release"
git push origin v1.0.0
Atencao: se voce alterar a versao exibida no sistema, atualize tambem a tag/release correspondente e publique a branch main antes ou junto da tag.

6.2 Problema comum: GitHub rejeitando arquivo grande

Se o GitHub recusar o push por arquivo acima de 100 MB, quase sempre a causa e algum arquivo local indevido no historico, como .venv ou binarios de dependencias.

Sinais comuns:

  • erro de arquivo acima de 100 MB
  • mensagens citando .venv, playwright/driver/node ou artefatos locais
  • push recusado mesmo apos apagar o arquivo do disco

Nesse caso:

  1. corrija o .gitignore
  2. remova o arquivo do rastreamento do Git
  3. se o blob ja entrou no historico, reescreva o historico antes de tentar novo push

7. O que o servidor Linux precisa ter

Para rodar este projeto em producao, o servidor precisa de:

  • Linux com acesso SSH
  • Git instalado
  • Docker instalado
  • Docker Compose ou plugin docker compose
  • Porta 8190 liberada no firewall ou exposta por proxy reverso

8. Instalacao basica no servidor Linux

Os comandos abaixo servem como base para distribuicoes Debian ou Ubuntu.

Atualizar pacotes

sudo apt update
sudo apt upgrade -y

Instalar Git

sudo apt install -y git

Instalar Docker

sudo apt install -y ca-certificates curl gnupg
sudo install -m 0755 -d /etc/apt/keyrings
curl -fsSL https://download.docker.com/linux/ubuntu/gpg | sudo gpg --dearmor -o /etc/apt/keyrings/docker.gpg
sudo chmod a+r /etc/apt/keyrings/docker.gpg

echo \
  "deb [arch=$(dpkg --print-architecture) signed-by=/etc/apt/keyrings/docker.gpg] https://download.docker.com/linux/ubuntu \
  $(. /etc/os-release && echo $VERSION_CODENAME) stable" | \
  sudo tee /etc/apt/sources.list.d/docker.list > /dev/null

sudo apt update
sudo apt install -y docker-ce docker-ce-cli containerd.io docker-buildx-plugin docker-compose-plugin

Permitir usar Docker sem sudo

sudo usermod -aG docker $USER
newgrp docker

Teste:

docker --version
docker compose version
Se seu servidor usar o binario legado: use docker-compose no lugar de docker compose.

9. Clonar o projeto no servidor

Crie uma pasta de trabalho e clone o repositório.

mkdir -p ~/projetos
cd ~/projetos
git clone https://github.com/sua-empresa/autocontrollers.git
cd autocontrollers

Se o repositório for privado, use a estrategia de autenticacao da empresa, como token ou chave SSH.

10. Preparar o arquivo de ambiente do servidor

O projeto possui um modelo pronto:

.env.server.example

Crie o arquivo real do servidor com base nele.

Linux

cp .env.server.example .env.server

Depois edite o arquivo:

nano .env.server

Exemplo de valores esperados:

DJANGO_SECRET_KEY=sua-chave-secreta-forte
DJANGO_DEBUG=False
DJANGO_ALLOWED_HOSTS=seu-dominio.com,IP_DO_SERVIDOR
DJANGO_CSRF_TRUSTED_ORIGINS=https://seu-dominio.com
DJANGO_SUPERUSER_USERNAME=admin
DJANGO_SUPERUSER_EMAIL=admin@empresa.com
DJANGO_SUPERUSER_PASSWORD=troque-esta-senha
DATABASE_URL=mysql://autocontrollers:senha-forte@db:3306/autocontrollers
AUTOMATION_SCHEDULER_ENABLED=0
AUTOMATION_WORKER_POLL_INTERVAL=5
Atencao: nunca suba .env.server para o GitHub. Ele deve existir apenas no servidor.

11. Ajustar o docker-compose para o servidor

Confirme se o docker-compose.yml esta lendo o arquivo de ambiente correto. O fluxo ideal em producao e usar:

.env.server

Se preferir, voce tambem pode exportar as variaveis manualmente antes do start.

12. Subir o sistema no servidor

Na raiz do projeto, rode o build e start dos containers.

Com plugin novo

docker compose up --build -d

Com binario legado

docker-compose up --build -d

Esse comando sobe:

  • db com MySQL
  • web com Django + Gunicorn
  • worker para consumir a fila

13. O que acontece na primeira subida

  • o MySQL inicializa o banco
  • o container web roda migrate
  • o container web roda collectstatic
  • o superusuario inicial e criado ou atualizado
  • o worker inicia e passa a monitorar a fila

14. Como validar que subiu corretamente

Ver containers ativos

docker compose ps

Ver logs da aplicacao

docker compose logs -f web

Ver logs do worker

docker compose logs -f worker

Ver logs do banco

docker compose logs -f db

Se estiver usando o binario antigo, troque docker compose por docker-compose.

Validar no navegador

http://IP_DO_SERVIDOR:8190

ou com dominio/proxy:

https://seu-dominio.com

15. Login inicial do Django

O projeto esta preparado para criar um superusuario inicial automaticamente com base nas variaveis abaixo:

  • DJANGO_SUPERUSER_USERNAME
  • DJANGO_SUPERUSER_EMAIL
  • DJANGO_SUPERUSER_PASSWORD

Depois do primeiro acesso, troque a senha imediatamente.

16. Atualizar o projeto no servidor depois de novas alteracoes

Quando houver novas alteracoes no GitHub, o fluxo no servidor normalmente sera:

cd ~/projetos/autocontrollers
git pull origin main
docker compose up --build -d

Para acompanhar o restart:

docker compose logs -f web
docker compose logs -f worker

17. Parar ou reiniciar o sistema

Parar

docker compose down

Reiniciar mantendo volumes

docker compose down
docker compose up --build -d

Recriar tudo, inclusive volumes

docker compose down -v
docker compose up --build -d
Atencao: o comando com -v remove volumes. Use com cuidado, porque ele apaga banco e arquivos persistidos dos volumes Docker se voce nao tiver backup.

18. Firewall e exposicao da porta

Se for acessar diretamente a porta da aplicacao, libere a 8190 no firewall.

Exemplo com UFW

sudo ufw allow 8190/tcp
sudo ufw status

Em ambiente corporativo, o mais recomendado e usar um proxy reverso como Nginx, com HTTPS e dominio.

19. Exemplo de proxy reverso com Nginx

Exemplo simples de bloco de servidor:

server {
    listen 80;
    server_name seu-dominio.com;

    location / {
        proxy_pass http://127.0.0.1:8190;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
}

Depois de configurar o Nginx:

sudo nginx -t
sudo systemctl reload nginx

20. Checklist de producao

  • o repositório esta no GitHub
  • o servidor possui Docker e Git
  • o arquivo .env.server foi criado
  • DJANGO_SECRET_KEY foi trocada
  • a senha do admin inicial foi trocada
  • DJANGO_DEBUG=False esta definido
  • ALLOWED_HOSTS esta correto
  • a porta 8190 ou o dominio esta acessivel
  • os containers web, worker e db estao ativos
  • o login no sistema funciona
  • o worker esta processando fila normalmente
